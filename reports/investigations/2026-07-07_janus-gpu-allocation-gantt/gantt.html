Root Cause: Device Plugin / DRA Resource Accounting Conflict

Both nvidia-device-plugin (publishing nvidia.com/gpu as extended resources) and composite-dra-driver (publishing composite.dra.io/gpu-nic-pair as DRA ResourceSlices) advertise the same physical GPUs on each node.

When nathans' pods consumed GPUs via nvidia.com/gpu, the device plugin decremented the node's allocatable count — but the composite DRA driver's ResourceSlices still showed 8 available gpu-nic-pair devices per node.

Key Finding: mkm Was FREE But DRA Still Rejected It

mkm had 0 device-plugin GPUs allocated from 13:41–19:44 UTC (6 hours!). The composite DRA ResourceSlice correctly published 8 gpu-nic-pair devices on mkm. Yet michey's DRA pod targeting mkm at 19:09 still got "Insufficient composite.dra/gpu-nic-pair."

Switching to nvidia.com/gpu device plugin immediately succeeded on the same node (mkm) at 19:44 — proving GPUs were physically free.

Root Cause Hypotheses

The composite DRA driver's template-reconciler (webhook) was continuously failing with RBAC errors — it could not list pods in glm-5-pp (michey's namespace) or default. This every-5-minute error (reconciler.go:66) may have:

  • Prevented cleanup of stale ResourceClaims from prior composite DRA test runs
  • Left the scheduler's in-memory DRA allocation table with phantom device reservations
  • Blocked the webhook from properly reconciling ResourceClaimTemplates in glm-5-pp

Evidence: Stale smoke-1-pair-composite-gpu-nic-pair ResourceClaimTemplate exists in default namespace since Jun 29 — the reconciler can't clean it up due to the same RBAC error.

Fix: (1) Grant composite-dra-driver ServiceAccount pod list/watch permissions in glm-5-pp and default namespaces. (2) Clean up stale ResourceClaimTemplates in default. (3) All users use the same GPU allocation mechanism to avoid cross-mechanism conflicts.
