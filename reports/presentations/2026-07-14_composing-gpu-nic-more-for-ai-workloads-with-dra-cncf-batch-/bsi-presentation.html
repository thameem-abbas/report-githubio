CHALLENGE 1

Multi-Device Allocation Is Manual

GPU and NIC are managed by separate DRA drivers. Pairing them by topology requires
per-rail ResourceClaimTemplates with hardcoded CEL selectors and routing config.
Note: generic attributes like pcieRoot and numaNode are being standardized across drivers — we already use these for pairing.

Source: openshift-psap/ibmcloud-roce-dra-net-setup 

# gpu-nic-rail0-template.yaml  (1 of 4 files)
apiVersion: resource.k8s.io/v1
kind: ResourceClaimTemplate
metadata:
  name: gpu-nic-rail0-template
spec:
  spec:
    devices:
      requests:
      - name: gpu
        exactly:
          deviceClassName: gpu.nvidia.com
          allocationMode: ExactCount
          count: 1
      - name: nic
        exactly:
          deviceClassName: dranet
          allocationMode: ExactCount
          count: 1
          selectors:
          - cel:
              expression: >-
                device.attributes["dra.net"]
                  .rdma == true
                && device.attributes["dra.net"]
                  .ipv4.startsWith("10.0.")
      constraints:
      - requests: [gpu, nic]
        matchAttribute:
          resource.kubernetes.io/pcieRoot
      # ⚠ Per-rail routing — hardcoded
      config:
      - requests: [nic]
        opaque:
          driver: dra.net
          parameters:
            interface:
              mtu: 9000
              name: net1
            routes:
            - destination: 10.0.0.0/16
              scope: 253
              table: 100
            - destination: 10.1.0.0/16
              gateway: 10.0.0.1
            - destination: 10.2.0.0/16
              gateway: 10.0.0.1
            - destination: 10.3.0.0/16
              gateway: 10.0.0.1
            # ... routes to all 8 rails
            - destination: 0.0.0.0/0
              gateway: 10.0.0.1
              table: 100
            rules:
            - priority: 32765
              source: 10.0.0.0/16
              table: 100

CHALLENGE 2

Per-Device Config for Sub-Node Pods

Inference serving, fine-tuning, and smaller batch jobs share a node.
Each pod gets a subset of GPU-NIC pairs — not the whole node.

Pod A gets rails 0–3 with gateways 10.0.0.1–10.3.0.1
Pod B gets rails 4–7 with gateways 10.4.0.1–10.7.0.1

?
which rails a pod gets
depends on scheduler placement
1
pod template
with static config

Per-device routing config depends on which devices are allocated.
That's only known after scheduling — too late for the pod template.

CHALLENGE 2

Two Bad Options for Sub-Node Allocation

Different workloads need 1, 2, 4, or 8 GPU-NIC pairs per pod

Option A: Pre-build templates per size

SizeTemplates
1 pair/pod8 (one per rail)
2 pairs/pod4 (rail0–1, rail2–3, …)
4 pairs/pod2 (rail0–3, rail4–7)
8 pairs/pod1 (all rails)

✗  15 templates per cluster
✗  Workload controller must assign
  the right template per replica
✗  Pods get pinned to specific rails

Option B: Keep 8 base templates (1 per rail)

Pod spec references N of the 8 rail templates.
Simpler template set, but:

✗  Workload controller must pick
  which rails each replica gets
✗  Single pod template can't differentiate —
  every replica gets the same rails
✗  Added complexity at the workload layer
  to handle per-replica rail assignment

Both options push infrastructure concerns into workload orchestration.

CHALLENGE 3

Scheduler Bypass Doesn't Scale

Prior approach: admission webhook scanned ResourceSlices,
picked nodes and devices, pinned pods via nodeAffinity

✗  Scheduler doesn't know what the webhook decided

✗  Can't optimize across workloads — preemption and rebalancing break

✗  Multi-tenant scheduling can't manage resources it doesn't control

✓  Goal: return device allocation authority to the K8s scheduler

CORE PATTERN

Shadow Claims: Pure Delegation

What we do

  • Create real ResourceClaims ("shadow claims")
  • Set status.allocation ourselves
    (normally the scheduler's job)
  • Call underlying drivers via gRPC with these claims
  • ownerReference → composite claim for GC cascade

What underlying drivers see

  • A normally-allocated ResourceClaim
  • kubeletplugin.Helper checks:
    ✓ claim exists
    ✓ allocation != nil
    ✓ UID matches
  • NOT checked: pod.spec presence,
    allocation provenance
Zero underlying driver code. Pure delegation via the existing DRA API.

DESIGN

Per-Device Config at Prepare Time

Opaque device params resolved via Go templates — not in the pod template

# GPU params — matched by product name
gpu.nvidia.com:
  entries:
  - match:
      "nvidia.com/productName":
        exact: "H100 80GB HBM3"
    values:
      ComputeMode: "DEFAULT"
  params: |
    {"computeMode": "{{.ComputeMode}}"}
# NIC params — matched by IP subnet
dra-net:
  entries:
  - match:
      "dra.net/ipv4": {prefix: "10.0.0."}
    values:
      Gateway: "10.0.0.1"
      TableID: "100"
  - match:
      "dra.net/ipv4": {prefix: "10.0.1."}
    values:
      Gateway: "10.0.1.1"
      TableID: "101"
  params: |
    {"gateway": "{{.Gateway}}",
     "tableID": {{.TableID}}}
At Prepare: gpu-2--nic-2 → GPU gets {"computeMode":"DEFAULT"}, NIC gets {"gateway":"10.0.2.1","tableID":102}
Template variables: {{.PairOrdinal}}, {{.NodeName}}, {{device "dra.net/ipv4"}} — per-device config without workload author involvement
DESIGN

Config, Not Code

sources:
  - name: gpu
    driver: "gpu.nvidia.com"
    forwardAttributes:
      - domain: "resource.kubernetes.io"
        attributes: [pcieRoot]
  - name: nic
    driver: "dra.net"
    forwardAttributes:
      - domain: "dra.net"
        attributes: [rdma, ipv4]
    filter:
      cel: 'device.attributes["dra.net"].rdma == true'

compositions:
  - name: "gpu-nic-pair"
    members: [{source: gpu, count: 1}, {source: nic, count: 1}]
    constraints:
      - type: matchAttribute
        attribute: "resource.kubernetes.io/pcieRoot"
Adding a new underlying driver is a YAML change — zero driver-specific code

COMMUNITY

What WG Device Management Said

Presented June 23, 2026 · wg-device-management/issues/54 · recording

Shadow claims — "Architecturally sound. DRA drivers don't check pod name or allocation provenance by design. Formalize if widely adopted."
Virtual ResourceSlices — "Valid extension of the DRA model. Would benefit from provenance metadata for safety."
Cross-pool exclusion — "Community interest in solving at DRA API level, not just at the driver layer."
Path forward — Short-term: ship as-is. Medium-term: kubernetes-sigs repo. Long-term: upstream API formalization.

ROADMAP

What's Next

Cross-composition device exclusion
Same physical device in multiple pools — need mutual exclusion. Exploring DRA API-level vs driver-level solutions. (#28)

Upstream formalization
Shadow claims as a reusable pattern. Derived ResourceSlice provenance. Based on WG Device Management guidance.

Broader device combinations
Config-driven design supports any DRA drivers. GPU+NIC is the first use case — storage, FPGA, accelerator compositions are possible.

Summary

1.  Batch training jobs need multi-device allocation (GPU+NIC)
  with per-device config — today this requires 50-line claims
  or scheduler bypass

2.  Composite DRA driver solves this within the existing DRA API
  Config-driven, validated at 318 Gb/s RDMA on real hardware

3.  Scheduling authority stays with the scheduler —
  no bypass, no side-channel, no admission webhook making placement decisions
